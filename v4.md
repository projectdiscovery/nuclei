# Nuclei v4 — Architectural Direction

This is a working design note, not a release plan. It captures the structural
changes that justify a major-version bump for nuclei — the ones that an
additive performance branch (like `feat-performance-improvements`) cannot
accomplish because they break the public surface, the on-disk format, the
SDK, or the template schema.

The criterion for inclusion is deliberately strict: a change earns a slot
here only if it is **breaking** *and* unlocks a step-change (5–10× perf, a
new capability, or removal of a load-bearing constraint). Incremental wins
belong in feature branches against `dev`.

## Why v4 (one paragraph)

v3 has accumulated load-bearing constraints that show up as friction every
time the engine is touched: the event model is a typed-erased
`map[string]interface{}` shared by every protocol, every matcher and every
extractor; templates are re-parsed from YAML on every startup; worker pools
are created per template; reflection sits on the HTTP success callback path;
the binary is monolithic at ~170 MB. Patching around these costs 1.2–2× at
best (which is what the perf branch achieved). Removing them — and shipping
the new contract that comes with their removal — is what v4 is for.

---

## #1 — Typed event model (kill `map[string]interface{}`)

**What changes.** Replace `output.InternalEvent = map[string]interface{}`
with a typed `Event` struct exposing the load-bearing fields directly:

```go
type Event struct {
    URL         *url.URL
    Method      string
    StatusCode  int
    Headers     http.Header
    Body        []byte
    Duration    time.Duration
    Request     []byte                 // raw dump
    Response    []byte                 // raw dump
    TemplateID  string
    TemplateInfo model.Info
    Matched     string
    Extracts    map[string]any         // user-defined extracts only
    Cache       *eventcache.Cache      // body decode cache (today: bodycache)
}
```

Matchers, extractors and DSL helpers gain typed accessors over this struct.
Untyped storage stays only for user-declared `Extracts`.

**Why it's breaking.** SDK consumers read `wrapped.InternalEvent[k]`
everywhere. DSL helper functions accept and return `interface{}`.
`MakeResultEvent` is part of every protocol. JSON output schema for
`-jsonl` would change.

**What it unlocks.**
- The replacer no longer needs `types.ToString(v)` per access.
- The body-decode cache stops being keyed by a magic string in a generic map
  (`__nuclei_bodycache`) and becomes a real field.
- `MergeMaps`, `MergeMapsMany`, and the entire reflect path collapse into
  typed merges of `Extracts`.
- Body becomes `[]byte` not `string`, eliminating the implicit copy at
  `string(body)`.
- Matchers can take a `*Event` instead of separately-constructed `corpus`
  and `data` arguments — the `getMatchPart` shim disappears.
- DSL gets typed function signatures (`StatusCode int`, not `interface{}`).

**Complexity.** High. Touches every protocol package and the DSL function
registry. Estimated as the single biggest refactor in v4.

**Prereqs.** None — this is a leaf change. Other v4 items get cleaner if
this lands first.

## #2 — Compiled template artifacts (Template IR + `nuclei compile`)

**What changes.** Introduce a binary template bundle format:

```
nuclei compile -t templates/ -o templates.nucbin
nuclei -bundle templates.nucbin -u https://...
```

The bundle contains:
- Pre-parsed template ASTs (no YAML at runtime).
- Pre-compiled regex, jq programs, DSL expressions.
- Pre-verified template signatures (with the verifier identity recorded).
- An index over IDs, tags, severity, protocols, paths.
- A version header (`bundle_format: 1`) and a `nuclei_version` constraint.

Today's directory-based loading remains supported as a fallback for local
development.

**Why it's breaking.** The bundle format becomes part of the public API.
Reproducibility expectations change — pinning a bundle hash pins behavior.
CI flows shift from "fetch templates repo at SHA" to "fetch versioned
bundle artifact."

**What it unlocks.**
- Cold start collapses from `O(N × YAML parse + verify + compile)` to
  `mmap + version check`. Expect 10–50× on large catalogs.
- Bundles can be signed once and distributed via PDCP / OCI / S3.
- Long-lived nuclei services (DAST server, scheduled scanners) stop paying
  the parse cost on every spawn.
- Deterministic, hash-pinned scans become trivial — bundle hash + target
  list = full provenance.

**Complexity.** Medium-high. The IR design is the hard part; the rest is
serialization plumbing.

**Prereqs.** Cleaner with #1 (typed event lets the IR avoid embedding any
`interface{}` placeholders for runtime types).

## #3 — Persistent work-stealing scheduler

**What changes.** Replace the per-template worker pool model in
`pkg/core/executors.go` with one global scheduler over typed task tuples:

```go
type Task struct {
    Template *Template
    Target   *Target
    Payload  *Payload
    Cluster  ClusterID            // for cross-template request dedup
}
```

The scheduler has work-stealing queues per worker, adaptive concurrency
tied to observed RTT and CPU load, first-class backpressure with bounded
queue depth, and a single shared HTTP client pool.

**Why it's breaking.** Resume semantics change (the `ResumeCfg.InFlight`
bookkeeping is per-template today). The `Engine` API exposed via
`pkg/core/` is not stable across this change. SPM (stop-at-first-match)
handlers move from per-request to per-cluster.

**What it unlocks.**
- Cross-template request dedup beyond opt-in clustering.
- Cache locality on hot HTTP clients (no more pool teardown per template).
- Tail-latency-aware throttling (slow targets get fewer workers).
- The buffered-tasks-channel patch from `feat-performance-improvements`
  becomes a non-event — backpressure is built in.

**Complexity.** High. Engine rewrite.

**Prereqs.** Easier with #1 (typed event makes task carrying simpler).
Composes well with #2 (bundle gives the scheduler a static shape to plan
against).

## #4 — Streaming I/O end-to-end

**What changes.**
- **Input:** target list streamed reservoir-style from stdin/file/socket;
  no full pre-load. Memory cost decoupled from `wc -l targets`.
- **Output:** lock-free SPSC ring per writer; on Linux, optional io_uring
  backend for the file writer. Today's `MultiWriter` fan-out (already added
  in `feat-performance-improvements`) becomes redundant.
- **Bodies:** chunked-body matchers where the operator allows it (regex,
  word, binary). HTML/XML/JSON still need full body for parsing.
- **Project cache:** drop LevelDB. The current goroutine dump shows 5+ live
  goroutines (`tCompaction`, `mCompaction`, `mpoolDrain`, buffer pool drain)
  for the entire run, even when nothing is stored. Replace with `bbolt`
  (already a transitive dep) or a tiny purpose-built KV.

**Why it's breaking.** Project file format changes (different KV backend).
Result ordering guarantees may change (lock-free rings break strict order).
Writer plugins must adopt the new interface.

**What it unlocks.**
- Order-of-magnitude reduction in resident memory on big scans.
- Removal of an entire class of "engine is busy doing GC on goleveldb"
  background noise.

**Complexity.** Medium. Mostly mechanical once the writer interface
changes.

**Prereqs.** None.

## #5 — WASM extension ABI; demote Goja

**What changes.** Add a wasmtime/wazero ABI for matchers, extractors,
post-processors, and (eventually) protocols. Templates can declare a
WASM-backed step:

```yaml
matchers:
  - type: wasm
    module: classifiers/login-page.wasm:v1
    params: {min_score: 0.85}
```

Goja remains as a compatibility shim for `code:` / `javascript:` templates
with deprecated status. New templates target WASM.

**Why it's breaking.** Sandbox model changes. The implicit "JS code can
shell out to anything Goja exposes" contract is replaced by an explicit
capability list per WASM module. Some `code:` templates won't be portable.

**What it unlocks.**
- Deterministic CPU/memory limits per extension call.
- Extensions in any language that targets WASM (Rust, Zig, AssemblyScript,
  TinyGo).
- A safe place to run inference modules (see #8).
- Removes a class of "Goja crashes the runtime" tickets.

**Complexity.** Medium. Wasmtime/wazero embedding is well-understood;
designing the ABI surface is the design work.

**Prereqs.** None directly; pairs naturally with #6 (modular protocols).

## #6 — Modular protocols, slim default binary

**What changes.** The default `nuclei` binary ships only HTTP, DNS, TCP,
SSL — the 90%-of-users core. Headless (Chrome deps), Code, JavaScript,
Fuzz, Network-extras, WebSocket, WHOIS, File become separately-distributed
modules loaded at runtime.

Module loading uses the WASM ABI from #5 where possible (lighter weight,
sandboxed) or a minimal Go plugin protocol over Unix-domain RPC for
modules that need full Go-runtime access (today: headless).

**Why it's breaking.** Installation changes (`nuclei install headless`).
Templates referencing missing protocols error explicitly instead of
silently working. Distribution becomes per-module versioned.

**What it unlocks.**
- Default binary drops from ~170 MB to ~30–40 MB.
- Independent release cadence per protocol.
- Smaller blast radius for protocol-specific CVEs in dependencies.
- Container images shrink significantly (Chrome alone is the bulk of the
  headless bundle).

**Complexity.** Medium-high. The plugin loader and the module distribution
infrastructure are the work.

**Prereqs.** #5 makes most of this trivial; some protocols (headless)
likely need the Go-plugin escape hatch.

## #7 — Explicit template schema versioning

**What changes.** Promote `template-schema:` to a load-bearing field. The
v4 engine compiles v3 templates through a translation layer to v4 IR.
Schema evolution after v4 follows a documented deprecation process.

```yaml
id: foo
template-schema: 4
info: {...}
http: [...]
```

**Why it's breaking.** Sets the precedent that template schema and engine
version are independently versioned. Some v3-only fields may not have v4
equivalents and require flagging.

**What it unlocks.**
- Users can upgrade nuclei without a flag-day for the entire templates
  repo.
- Schema can move forward without backward-compat ratchets baked into the
  parser.

**Complexity.** Low for the field itself; medium for the v3→v4 translator.

**Prereqs.** Has to land *before* any other schema-affecting change in v4.

## #8 — Optional onboard inference

**Status: addon, not core.** Inference is a v4-era feature because it adds
non-deterministic-by-default behavior to a tool whose contract has always
been deterministic. Treat it as opt-in everywhere, locally-runnable by
default, and architected so a scan run without it is unchanged from today.

### Where it earns its keep

1. **False-positive triage as a post-matcher.** A small on-device
   classifier (ONNX MiniLM-class, ~30 MB) runs *only on already-matched
   responses*, scoring `(template_id, status_code, body_excerpt,
   header_subset) → likely_fp`. Doesn't change the truth output (matched
   stays matched); adds a confidence score and an optional
   `--fp-threshold` filter. Highest leverage, lowest risk.

2. **A new `ai:` matcher / extractor type.** Sits alongside
   `word`/`regex`/`dsl`/`xpath`:

   ```yaml
   matchers:
     - type: ai
       model: minilm-classifier:v1
       prompt: is_login_page
       threshold: 0.8
   ```

   Outputs are cached by `(model_version, prompt_hash, body_hash)`. The
   body-decode cache added in `feat-performance-improvements` is the right
   place for this — extend it with a `Classify(label, body) (bool,
   float32, error)` slot so multiple `ai:` matchers against the same
   response share one inference call.

3. **Natural-language template search and synthesis as tooling, not
   runtime.** New CLI verbs:
   - `nuclei explain <template-id>` — readable summary.
   - `nuclei find "reflected XSS via Referer header"` — embedding search
     over the catalog.
   - `nuclei draft -from-response request.txt response.txt` — generate
     template skeleton.

   Pure tooling; doesn't touch the scan engine.

4. **Context-aware fuzz payload mutation.** A small generator conditioned
   on parameter name/type/observed reflection produces candidates that
   complement the static dictionary fuzzing. Capped per request to avoid
   throughput collapse.

5. **Tech-stack inference for unknown fingerprints.** A cheap classifier
   over headers + body fingerprints enriches `tags`/`metadata` when
   wappalyzergo doesn't know the stack. Feeds into `-as` (auto-scan).

### What makes it non-trivial in nuclei specifically

- **Determinism is load-bearing.** Today you can re-run a scan and assert
  identical output — CI, audit, regression all rely on it. Inference
  breaks this unless we pin: model identity, weights hash, temperature=0,
  and result cache. Hard contract for v4: `(nuclei_version, model_pin,
  cached_response_bytes) → identical matcher verdict`. Anything else is
  opt-in via a clearly labeled flag, and the result event records that
  inference contributed.

- **Latency budget per response is microseconds.** A model invoked on
  every body would gut throughput. Two mitigations:
  - Run inference *only* on already-matched responses (FP triage), so
    scaling is `O(matches)` not `O(requests)`.
  - Cache by body hash so multi-template clusters hit the model once.

- **Body content is hostile input.** Prompt injection against a local
  model is a real concern; against a tool-using NL CLI it's worse. Treat
  model input as untrusted, never let response content control tool-use
  paths, and run inference inside the WASM sandbox from #5.

- **Binary size and dependency creep.** ONNX-runtime and llama.cpp via cgo
  are heavy. This is exactly why #6 (modular protocols) matters — `nuclei
  ai-pack` becomes an opt-in module, default binary stays slim.

- **Local vs. remote provider abstraction.** Same interface, two backends:
  - Embedded (default, deterministic, offline): ONNX/llama.cpp with
    version-pinned tiny models bundled into the `ai-pack` module.
  - Remote (opt-in, visible cost): PDCP-hosted, OpenAI-compatible, etc.
    Budget controls (`--ai-budget 1000-tokens`, `--ai-budget 0.05-USD`)
    and explicit per-template enablement.

### Honest limits

- **Don't ship a generative LLM in the default binary.** Embeddings + small
  classifiers, yes; a 7B-parameter model, no — that lives in
  `nuclei-ai-pack`.
- **Don't pitch CVE discovery via LLM-only matchers.** Inference is a
  triage and tooling layer over deterministic matchers, not a replacement
  for them. The "zero false positives" pitch collapses the moment a
  probabilistic matcher is the only signal for a finding.
- **Don't let inference results auto-mutate templates.** `nuclei draft` is
  a starting point for humans; auto-promotion is out of scope.

### How it fits the rest of v4

- Sits on top of #1 (typed event gives the model structured features
  instead of stringly-typed fields).
- Bakes model pins into #2 (compiled bundle records `model_version` so a
  scan is reproducible).
- Runs inside #5 (WASM ABI sandboxes the inference module and enforces
  capability limits).
- Distributed as an optional module per #6.

## Prioritization

If only two land, pick #1 and #2:

- **#1 (typed event)** clears the path for #3 (scheduler), #4 (streaming),
  #5 (WASM ABI), and #8 (inference) — every other item gets cheaper.
- **#2 (compiled artifacts)** delivers the most user-visible win on its
  own (cold start) and unlocks reproducible deployments + signed bundle
  distribution.

Sequencing for the rest:

```
   #7 (schema versioning) ─── must land first
                ↓
   #1 (typed event)       ─── biggest refactor
                ↓
   ┌──────────┴─────────┐
   #2 (IR/bundle)   #4 (streaming I/O)
        ↓                ↓
   #3 (scheduler)     #6 (modular protocols)
                          ↓
                       #5 (WASM ABI)
                          ↓
                       #8 (inference)
```

#5 → #8 is the canonical AI-as-sandboxed-module path. #6 can land before
#5 with a Go-plugin loader as a stop-gap.

## Non-goals (explicitly *not* v4-worthy)

- Replacing Goja with QuickJS / Boa / a different JS runtime — incremental.
- HTTP/3 support — protocol-level addition, fits in dev.
- Better workflow language — incremental.
- Better error messages — incremental.
- A Web UI — out of scope for the engine; belongs to a separate product.
- LLM-generated templates as a default behavior — see Honest limits in #8.

---

## Risks / open questions

- **#1 SDK churn.** Heaviest external impact. Need a clean migration shim
  for the `lib/` SDK consumers — probably an adapter that constructs a
  `*Event` from a `map[string]any` for one release, with a deprecation
  flag.
- **#2 signature model.** Today templates are signed individually. A
  bundle introduces a new signing question: per-template signatures
  carried in the bundle, or a single bundle signature? Probably both
  (per-template for provenance, bundle for distribution integrity).
- **#3 backpressure visibility.** A persistent scheduler hides per-template
  throughput from the user. Need first-class metrics (per-template QPS,
  pool occupancy) to keep operators happy.
- **#5 ABI stability.** Once external WASM modules ship, the ABI is
  effectively a public API. Versioning strategy must be specified before
  the first module is published.
- **#6 module distribution.** Where do modules live? PDCP hosting? OCI
  registry? Plain HTTPS download with manifests? Decision affects supply
  chain trust model.
- **#8 inference reproducibility.** Even with temperature=0 and pinned
  weights, inference engines occasionally diverge across CPU
  microarchitectures (different SIMD paths, different rounding). Need a
  bit-exact reference implementation or accept a documented
  "reproducible-modulo-architecture" caveat.

## What's already in flight (informs sequencing)

- **`feat-performance-improvements`** (this branch): 10 additive
  optimizations targeting v3 hot paths. Median 2.22× on a representative
  bench. Establishes the body-decode cache that #8 reuses, the typed
  `MergeMapsMany` paths that #1 generalizes, and the buffered-channel /
  concurrent-writer wins that #3 / #4 subsume entirely.
- **In-flight community PRs** (referenced during the perf pass): RFC 9111
  HTTP response cache, scan strategy improvements, additional protocol
  hardening. None of these block the v4 items above; most either slot
  into the v3 line or compose cleanly with v4.
